## Phase 1 — Fix the Logging Architecture (Two-Row Problem)

> **Root cause:** Each chat turn currently inserts **two separate DB rows** — one for the AI translation step and one for the SQL execution step — instead of one unified row per turn. This corrupts all analytics.

### What's happening now

```
User asks: "Total Customer With Name"

Row 1 → LogID: N    CompanyID: 0    UserQuestion: [ENTIRE SYSTEM PROMPT]
         ResponseType: AI_json       IsSuccess: 0    ErrorMessage: "AI returned json"

Row 2 → LogID: N+1  CompanyID: 10891  UserQuestion: "Total Customer With Name"
         ResponseType: AI_Success    SQLQuery: "SELECT..."    IsSuccess: 1
```

Two rows for one turn. Row 1 has CompanyID=0, contains the system prompt in UserQuestion, and is marked as failure even when the overall query succeeded. Row 1 is noise.

### Step 1.1 — Understand the fix model

Every chat turn must produce **exactly one row** in `AIChatLog`. That row captures the full lifecycle of the turn:

```
LogID        → GUID for this turn
CompanyID    → actual company (never 0)
UserQuestion → ONLY the plain human question string
SQLQuery     → the SQL generated by the LLM (populated after translation)
AIResponse   → ONLY the final formatted text shown to the user
ResponseType → final status: Fast | AI_Success | AI_NoData | AI_UnsafeSQL | AI_Exception | AI_NoSQL
ExecutionTime → total ms from question received to response returned
IsSuccess    → 1 if user got a real answer, 0 only on genuine failure
ErrorMessage → null on success, actual exception message on failure
CreatedDate  → set once, at insert time
```

### Step 1.2 — Locate the logging calls in `Nl2SqlService.cs`

Find every place that calls the log repository or inserts an `AIChatLog` record. There will be **at least two insert calls** — one inside the AI translation block and one inside the SQL execution block. You need to collapse them into one.

### Step 1.3 — Refactor to a single-row pattern

Replace the multiple insert calls with this pattern:

```csharp
// In Nl2SqlService.cs — ProcessQueryAsync or equivalent method

public async Task<string> ProcessQueryAsync(string userQuestion, int companyId)
{
    var sw = System.Diagnostics.Stopwatch.StartNew();

    // Declare log object ONCE at the top of the method
    var log = new AIChatLog
    {
        LogID       = Guid.NewGuid().ToString(),
        CompanyID   = companyId.ToString(),   // NEVER leave as 0
        UserQuestion = userQuestion,           // ONLY the human's text
        CreatedDate = DateTime.Now.ToString("yyyy-MM-dd HH:mm:ss"),
        IsSuccess   = "0"                      // default pessimistic, overwrite on success
    };

    try
    {
        // --- Stage 1: AI translates question to SQL ---
        string generatedSql = await TranslateToSql(userQuestion, companyId);
        log.SQLQuery = generatedSql;          // populate on the SAME object

        // --- Safety check ---
        if (!IsSafeSelect(generatedSql))
        {
            log.ResponseType = "AI_UnsafeSQL";
            log.AIResponse   = "⚠️ Unable to process safely.";
            log.ErrorMessage = "Blocked: non-SELECT or missing CompanyID filter";
            // fall through to finally block for single insert
            return log.AIResponse;
        }

        // --- Stage 2: Execute SQL ---
        var resultJson = ExecuteSql(generatedSql, companyId);

        // --- Stage 3: Format response ---
        string formatted = await FormatResponse(userQuestion, generatedSql, resultJson);

        log.AIResponse   = formatted;
        log.ResponseType = "AI_Success";
        log.IsSuccess    = "1";

        return formatted;
    }
    catch (Exception ex)
    {
        log.ResponseType = "AI_Exception";
        log.AIResponse   = "🤖 I'm unable to process that right now.";
        log.ErrorMessage = ex.Message;
        return log.AIResponse;
    }
    finally
    {
        // Single insert — always runs, captures the full turn
        sw.Stop();
        log.ExecutionTime = sw.ElapsedMilliseconds.ToString();
        _chatLogRepository.Insert(log);   // ONE insert, never more
    }
}
```

Search the entire solution for any other call to `AIChatLogRepository.Insert` or direct `db.AIChatLogs.Add(...)`. **Delete them all.** The `finally` block in step 1.3 is the only place a log row is ever written.

## Phase 2 — Fix the Raw API Response Stored in AIResponse

> **Root cause:** When the AI returns a response wrapped in markdown code fences (` ```json ... ``` `), the parser fails to strip them and falls through to a catch block that saves the **raw Anthropic HTTP response envelope** into `AIResponse` instead of the extracted SQL string.

### What's stored now (wrong)

```json
{"model":"claude-haiku-4-5-20251001","id":"msg_013G7...","content":[{"type":"text","text":"```json\n{\"sql\":\"SELECT...\"}\n```"}],"usage":{...}}
```

This entire JSON blob ends up in `AIResponse` and `UserQuestion`. The real extracted text was `{"sql":"SELECT..."}` which was inside `content[0].text` but wasn't reached because the markdown fence stripper wasn't in place.

### Step 2.1 — Locate `TranslateToSql` or equivalent method

Find the method that calls the AI API and tries to extract the SQL from the response. It will look roughly like:

```csharp
var rawResponse = await _httpClient.PostAsync(...);
string body = await rawResponse.Content.ReadAsStringAsync();
dynamic json = JsonConvert.DeserializeObject(body);
string text = json.choices[0].message.content; // Groq pattern
// OR
string text = json.content[0].text;             // Claude pattern
```

### Step 2.2 — Add a robust extraction + fence-stripping method

Add this utility method to the service class:

```csharp
private string ExtractTextFromApiResponse(string rawHttpBody)
{
    try
    {
        dynamic json = JsonConvert.DeserializeObject(rawHttpBody);

        // Try Groq/OpenAI format first
        string text = null;
        try { text = (string)json.choices[0].message.content; } catch { }

        // Try Anthropic/Claude format
        if (text == null)
            try { text = (string)json.content[0].text; } catch { }

        if (text == null)
            throw new Exception("Could not extract text from API response");

        return text;
    }
    catch (Exception ex)
    {
        throw new Exception($"API response parse failed. Raw (first 300 chars): {rawHttpBody.Substring(0, Math.Min(300, rawHttpBody.Length))}", ex);
    }
}

private string StripMarkdownFences(string text)
{
    if (string.IsNullOrWhiteSpace(text)) return text;

    // Strip ```json ... ``` or ``` ... ```
    text = text.Trim();
    if (text.StartsWith("```"))
    {
        int firstNewline = text.IndexOf('\n');
        if (firstNewline >= 0)
            text = text.Substring(firstNewline + 1);
        if (text.EndsWith("```"))
            text = text.Substring(0, text.Length - 3);
    }
    return text.Trim();
}

private string ExtractSqlFromJson(string jsonText)
{
    jsonText = StripMarkdownFences(jsonText);
    try
    {
        dynamic parsed = JsonConvert.DeserializeObject(jsonText);
        return (string)parsed.sql;
    }
    catch
    {
        // Last resort: if the LLM returned raw SQL without JSON wrapper
        if (jsonText.TrimStart().StartsWith("SELECT", StringComparison.OrdinalIgnoreCase))
            return jsonText.Trim();

        throw new Exception("AI returned no SQL query");
    }
}
```

### Step 2.3 — Wire them together in your translation call

```csharp
string rawHttpBody = await response.Content.ReadAsStringAsync();
string extractedText = ExtractTextFromApiResponse(rawHttpBody);
string generatedSql  = ExtractSqlFromJson(extractedText);
// generatedSql now contains only the clean SELECT statement
```

### Step 2.4 — Ensure AIResponse never stores the raw HTTP body

`AIResponse` must only ever contain the **final formatted string shown to the user**. In the Phase 1 pattern, `log.AIResponse` is set only after `FormatResponse(...)` completes. If an exception is thrown earlier, `log.AIResponse` is the user-facing error message. The raw HTTP body must never be assigned to `log.AIResponse`.

## Phase 3 — Fix the Sign Convention Bug in `ParseClosingBalance`

> **Root cause:** The code comment says "Credit positive, Debit negative" but the implementation is **inverted** — Credits are stored as negative and Debits as positive. Every aggregation query (total assets, total liabilities, P&L) returns wrong numbers.

### What's wrong now

```csharp
// CURRENT — WRONG
if (decimal.TryParse(numberPart, out decimal val))
    return isCredit ? -val : val;  // Credit = negative, Debit = positive (INVERTED)
```

### The accounting convention

In Tally and standard double-entry bookkeeping:
- **Credit (Cr)** = Liabilities, Equity, Revenue → these should be **positive** in your schema
- **Debit (Dr)** = Assets, Expenses → these should be **negative** in your schema (as documented)

The schema documentation states: *"Debits (Dr) are stored as negative decimals... Credits (Cr) are stored as positive decimals"*

### Step 3.1 — Fix in `TallyXmlParser.cs`

```csharp
public decimal ParseClosingBalance(string rawVal)
{
    if (string.IsNullOrWhiteSpace(rawVal)) return 0;
    string text = rawVal.Trim();

    bool isCredit = text.EndsWith("Cr", StringComparison.OrdinalIgnoreCase);
    bool isDebit  = text.EndsWith("Dr", StringComparison.OrdinalIgnoreCase);

    string numberPart = (isCredit || isDebit)
        ? text.Substring(0, text.Length - 2).Trim()
        : text;

    numberPart = numberPart.Replace(",", "");  // strip Indian number formatting

    if (decimal.TryParse(numberPart, System.Globalization.NumberStyles.Any,
        System.Globalization.CultureInfo.InvariantCulture, out decimal val))
    {
        // FIXED: Credit = positive (liabilities/equity), Debit = negative (assets)
        return isDebit ? -val : val;
    }

    return 0;
}
```

### Step 3.2 — Re-sync after fixing

After deploying this fix, you **must** trigger a full re-sync from Tally. All existing `TallyLedger.ClosingBalance` values in the database are stored with the wrong sign. The DELETE + INSERT on next sync will correct them automatically.

### Step 3.3 — Verify with a known ledger

Pick a ledger you know the balance of in Tally (e.g., "Cash"). Run:

```sql

```

Cash is an Asset (Debit nature), so `ClosingBalance` should be a **negative** number (e.g., `-50000.00`). If it was previously positive, the fix worked.

## Phase 4 — Fix the AI Generating Non-Existent Column Names

> **Root cause:** The LLM is generating SQL with column names that don't exist in the actual schema — confirmed in live logs: `Invalid column name 'CreatedDate'` for Customer table, and `The multi-part identifier "Invoice.InvoiceID" could not be bound`. The system prompt schema definition is incomplete or ambiguous.

### Confirmed failures from live logs

| LogID | Question | Error |
|---|---|---|
| 103 | "25 feb na data apo" | `The multi-part identifier "Invoice.InvoiceID" could not be bound` |
| 20058 | "today customer" | `Invalid column name 'CreatedDate'` |
| 20068 | "aaje ketla customer add kar ya" | `Invalid column name 'CreatedDate'` |

### Root cause analysis

The `Customer` table does **not** have a `CreatedDate` column (confirmed in the live schema). The LLM is hallucinating it when the user asks about "today's" or "new" customers. The schema definition in the system prompt does not list `Customer` creation date, but the LLM assumes it exists.

The `Invoice.InvoiceID` bind error is a JOIN alias problem — the LLM is using `Invoice.InvoiceID` in a subquery context where the outer alias isn't visible.

### Step 4.1 — Add negative constraints to the system prompt

In `Nl2SqlService.cs`, find the system prompt string that lists the table schema. Add these explicit negation rules inside the `SQL RULES` section:

```
=====================
COLUMN EXISTENCE RULES (CRITICAL)
=====================
- Customer table has NO CreatedDate, NO AddedDate, NO JoinDate column
- Customer table has NO date column of any kind
- If user asks about "new customers today" or "customers added today", return: { "sql": "NO_DATE_COLUMN" }
- Invoice table has InvoiceDate (not CreatedDate, not AddedDate)
- Payment table has PayDate (not CreatedDate, not PaymentDate)
- NEVER use a column name not explicitly listed in the TABLES section above
- If uncertain about a column name, use only columns from the TABLES list
```

### Step 4.2 — Handle the `NO_DATE_COLUMN` sentinel in code

In your `ExtractSqlFromJson` method (added in Phase 2), add:

```csharp
private string ExtractSqlFromJson(string jsonText)
{
    jsonText = StripMarkdownFences(jsonText);
    try
    {
        dynamic parsed = JsonConvert.DeserializeObject(jsonText);
        string sql = (string)parsed.sql;

        // Handle sentinel values the LLM returns for impossible queries
        if (sql == "NO_DATE_COLUMN" || sql == "NO_DATA" || sql == "UNSUPPORTED")
            throw new Exception("AI returned no SQL query");  // handled as AI_NoSQL

        return sql;
    }
    catch (Exception ex) when (!ex.Message.Contains("AI returned"))
    {
        if (jsonText.TrimStart().StartsWith("SELECT", StringComparison.OrdinalIgnoreCase))
            return jsonText.Trim();
        throw new Exception("AI returned no SQL query");
    }
}
```

### Step 4.3 — Fix the Invoice JOIN binding error

The `Invoice.InvoiceID could not be bound` error occurs when the LLM writes a subquery that references `Invoice.InvoiceID` outside the scope where the `Invoice` alias is defined. Add this to the system prompt:

```
=====================
SUBQUERY RULES
=====================
- NEVER reference a table alias from the outer query inside a subquery
- In subqueries, always re-JOIN the required table
- Prefer LEFT JOIN in the main query over subqueries wherever possible
- If you need Invoice data and Payment data together, use LEFT JOIN PaymentMap ON PaymentMap.InvoiceID = Invoice.InvoiceID directly — do not nest
```

## Phase 6 — Fix the Bulk Insert Performance in `TallyRelationalRepository`

> **Root cause:** The sync upsert loop calls `ctx.TallyLedgers.Add(ledger)` one record at a time inside a loop and calls `ctx.SaveChanges()` once at the end. For large companies, EF6 will generate N individual `INSERT` statements executed sequentially, holding a table lock throughout.

### What's happening now

```csharp
foreach (var ledger in ledgers)          // e.g. 5,000 ledgers
{
    ledger.LedgerID  = Guid.NewGuid().ToString();
    ledger.CompanyID = companyId;
    ledger.SyncedAt  = DateTime.Now;
    ctx.TallyLedgers.Add(ledger);        // queued in EF change tracker
}
ctx.SaveChanges();                       // fires 5,000 INSERT statements
```

EF6's change tracker holds all 5,000 objects in memory and generates 5,000 round-trips to SQL Server.

### Step 6.1 — Add `SqlBulkCopy` for the insert path

Add a new method `BulkInsertLedgers` in `TallyRelationalRepository.cs`. This requires a raw `SqlConnection`, not EF:

```csharp
using System.Data;
using System.Data.SqlClient;

private void BulkInsertLedgers(List<TallyLedger> ledgers, int companyId, SqlConnection conn, SqlTransaction tx)
{
    var table = new DataTable();
    table.Columns.Add("LedgerID",       typeof(string));
    table.Columns.Add("CompanyID",      typeof(int));
    table.Columns.Add("Name",           typeof(string));
    table.Columns.Add("Parent",         typeof(string));
    table.Columns.Add("ClosingBalance", typeof(decimal));
    table.Columns.Add("SyncedAt",       typeof(DateTime));

    foreach (var ledger in ledgers)
    {
        table.Rows.Add(
            Guid.NewGuid().ToString(),
            companyId,
            ledger.Name,
            ledger.Parent,
            ledger.ClosingBalance,
            DateTime.Now
        );
    }

    using (var bulk = new SqlBulkCopy(conn, SqlBulkCopyOptions.Default, tx))
    {
        bulk.DestinationTableName = "TallyLedger";
        bulk.BatchSize            = 500;
        bulk.WriteToServer(table);
    }
}
```

### Step 6.2 — Refactor the sync method to use raw connection + bulk copy

```csharp
public void UpsertLedgers(List<TallyLedger> ledgers, int companyId)
{
    string connStr = System.Configuration.ConfigurationManager
        .ConnectionStrings["InsidashTallyConnection"].ConnectionString;

    using (var conn = new SqlConnection(connStr))
    {
        conn.Open();
        using (var tx = conn.BeginTransaction())
        {
            try
            {
                // Step 1: Delete existing rows for this company
                using (var cmd = new SqlCommand(
                    "DELETE FROM TallyLedger WHERE CompanyID = @cid", conn, tx))
                {
                    cmd.Parameters.AddWithValue("@cid", companyId);
                    cmd.ExecuteNonQuery();
                }

                // Step 2: Bulk insert new rows
                BulkInsertLedgers(ledgers, companyId, conn, tx);

                tx.Commit();
            }
            catch
            {
                tx.Rollback();
                throw;
            }
        }
    }
}
```

Apply the same pattern to `UpsertVouchers` for the `TallyVoucher` table.

## Phase 7 — Fix the SQL Safety Check

> **Context:** `AI_UnsafeSQL` is already being caught (5 confirmed cases in live logs — DELETE, INSERT, and an unrelated Gujarati question got blocked). The safety check exists but needs hardening. One confirmed case (`wrong data ave che sacha data kevi rite apu` — "how to give correct data") was incorrectly blocked as unsafe SQL. The check is too aggressive.

### Step 7.1 — Review what the current check does

Find `IsSafeSelect` or equivalent validation. It likely does a simple `!sql.TrimStart().StartsWith("SELECT")`. The Gujarati question was probably returning a non-SQL response from the AI (a conversational reply) that was then being run through the SQL safety check unnecessarily.

### Step 7.2 — Replace with a specific allowlist check

```csharp
private bool IsSafeSelect(string sql)
{
    if (string.IsNullOrWhiteSpace(sql)) return false;

    string upper = sql.Trim().ToUpperInvariant();

    // Must start with SELECT
    if (!upper.StartsWith("SELECT")) return false;

    // Must not contain any write/DDL keywords
    var forbidden = new[] { "INSERT", "UPDATE", "DELETE", "DROP", "TRUNCATE",
                             "ALTER", "CREATE", "EXEC", "EXECUTE", "SP_",
                             "XP_", "OPENROWSET", "OPENDATASOURCE" };
    foreach (var word in forbidden)
    {
        // Use word boundary check to avoid false positives (e.g. "SELECTED")
        if (System.Text.RegularExpressions.Regex.IsMatch(upper, $@"\b{word}\b"))
            return false;
    }

    // Must include CompanyID filter (prevents cross-tenant data leaks)
    if (!upper.Contains("COMPANYID")) return false;

    return true;
}
```

### Step 7.3 — Add a pre-check for non-SQL AI responses

Before running `IsSafeSelect`, check if the AI returned a conversational response instead of SQL. This prevents legitimate non-SQL responses from being logged as `AI_UnsafeSQL`:

```csharp
// In TranslateToSql, after ExtractSqlFromJson throws for non-SQL:
// The exception is caught as AI_NoSQL, not AI_UnsafeSQL.
// Only call IsSafeSelect when you have a string that looks like SQL.
// ExtractSqlFromJson already throws "AI returned no SQL query" for non-SQL responses.
// So by the time IsSafeSelect is called, you already have a SELECT statement or similar.
```
